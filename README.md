# Autonomous Incident Response

An AI system that responds to production outages the moment an alert fires. Alert in ā†’ Slack brief and automated remediation out in seconds, with a post-mortem generated the moment the incident is marked resolved.

## What happens on a page

1. **Webhook hits `POST /alerts`** with a Datadog/PagerDuty/generic-style JSON payload.
2. **Auth**: shared token (`X-Webhook-Token`) *or* HMAC signature (`X-Datadog-Signature` / `X-PagerDuty-Signature` / `X-Webhook-Signature`) ā€” any one credential is sufficient.
3. **Rate limit**: sliding-window in-memory limiter, keyed by `(client_ip, service)`.
4. **Dedup**: alert is fingerprinted by `(service, metric, severity, 15-min bucket)`. Repeats in the same bucket attach as timeline entries on the existing incident instead of opening a new one.
5. `/alerts` returns `202 {"status": "accepted", "incident_id": "inc-..."}` in single-digit ms and enqueues the alert to a background worker.
6. **Worker runs the orchestrator** which does three things in parallel via `asyncio.gather`:
   - **Triage agent** ā€” pulls the last 90 min of commits, asks Claude to rank up to 3 suspects by timing, files, message, and diff size, and returns confidence + reasoning per suspect.
   - **Runbook agent** ā€” asks Claude to pick the best-matching runbook from `./runbooks/*.md` (title + tags + first body line, no full-content shipping to save tokens); returns `None` below a 0.3 confidence floor to avoid confidently-wrong matches.
   - **Impact agent** ā€” pulls error-rate + request-rate series + active-user count, asks Claude for an integer affected-user estimate + peak error rate.
7. **Slack brief** is composed (deterministic template, no LLM) and posted to the incidents channel.
8. **Runbook remediation** ā€” if the matched runbook declares a `## Automated actions` JSON block, allow-listed `auto: true` steps run through the shell executor and results post as a threaded reply. Default `REMEDIATION_MODE=mock` never touches the system.
9. **Incident state** is persisted to SQLite after every step, so a restart mid-flow doesn't lose anything.

When someone hits `POST /alerts/{id}/resolve`:

- A **blameless post-mortem** is generated by Claude (summary ā†’ impact ā†’ timeline ā†’ root cause ā†’ detection ā†’ mitigation ā†’ action items with P0/P1/P2 ā†’ lessons) and saved to `./postmortems/YYYY-MM-DD-inc-*.md`.
- A threaded Slack reply links the post-mortem back to the original brief.

External calls (Anthropic, GitHub REST, Slack webhook, Datadog) are wrapped in exponential-backoff-with-jitter retry. Every integration has a `Mock*` and a real client behind the same interface ā€” the whole flow runs offline and in tests.

## Endpoints

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/alerts` | Enqueue an incident. Returns `202 {status, incident_id}`. |
| `POST` | `/alerts/{id}/resolve` | Mark resolved; generate post-mortem; thread reply. Returns the full incident. |
| `GET`  | `/incidents/{id}` | Current state (status, triage report, timeline, post-mortem path). |
| `GET`  | `/healthz` | Liveness. |
| `GET`  | `/readyz` | Liveness + current worker queue depth. |

## Quick start

```bash
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"           # or ".[dev,otel]" for OpenTelemetry export
cp .env.example .env              # set ANTHROPIC_API_KEY at minimum
uvicorn incident_response.main:create_app --factory --reload --port 8080
```

Fire a test alert:

```bash
curl -X POST http://localhost:8080/alerts \
  -H "x-webhook-token: change-me" \
  -H "content-type: application/json" \
  -d '{
    "id": "ddg-9273",
    "title": "Checkout 5xx > 5%",
    "service": "checkout",
    "severity": "sev2",
    "triggered_at": "2026-07-02T21:05:00+00:00",
    "metric": "http.error_rate",
    "value": 0.184
  }'
# ā†’ 202 {"status":"accepted","incident_id":"inc-ddg-9273"}
```

Fetch the incident once triage finishes (usually 2ā€“5 s):

```bash
curl http://localhost:8080/incidents/inc-ddg-9273
```

Resolve and auto-generate the post-mortem:

```bash
curl -X POST http://localhost:8080/alerts/inc-ddg-9273/resolve \
  -H "x-webhook-token: change-me" \
  -H "content-type: application/json" \
  -d '{"resolution_note": "rolled back a1b2c3d"}'
```

Post-mortem lands at `./postmortems/YYYY-MM-DD-inc-ddg-9273.md`.

## Wiring to real systems

Flip mode env vars in `.env`:

| Env var | Modes | Notes |
|---|---|---|
| `GITHUB_MODE` | `mock`, `rest` | `rest` needs `GITHUB_TOKEN` + `GITHUB_REPO` |
| `SLACK_MODE` | `mock`, `webhook`, `bot` | `webhook` needs `SLACK_WEBHOOK_URL`; `bot` needs `SLACK_BOT_TOKEN` and enables streaming brief updates via `chat.update` |
| `METRICS_MODE` | `mock`, `datadog` | `datadog` needs `DATADOG_API_KEY` + `DATADOG_APP_KEY` |
| `REMEDIATION_MODE` | `mock`, `shell` | `shell` runs only steps in `REMEDIATION_ALLOWED_COMMANDS` with `auto: true` |

Adding another provider (GitLab, Prometheus, ServiceNow) means one new class in `integrations/` and one line in the matching `build_*_client` factory.

## Runbook library

Markdown in `./runbooks/`. Frontmatter drives search; an optional JSON block declares remediation steps.

````markdown
---
title: Checkout service elevated error rate
tags: [checkout, http_5xx]
---

## First actions
1. Confirm the alert in Datadog.
2. Check the last 3 deploys...

## Automated actions
```json
[
  {"name": "flip pricing cache off", "command": "feature-flag set checkout.pricing_cache off", "auto": true},
  {"name": "rollback last deploy",   "command": "deploy rollback checkout --confirm"}
]
```
````

- Only steps with `"auto": true` **and** a first token present in `REMEDIATION_ALLOWED_COMMANDS` ever run.
- Everything else is reported as `skipped_not_auto` / `skipped_not_allowed` ā€” full audit trail, no silent skips.
- Default `REMEDIATION_MODE=mock` always returns `dry_run` ā€” safe to point at a runbook library without opting in first.
- Results (executed, dry_run, skipped, failed) are posted as a threaded Slack reply on the incident brief.

## Production hardening

| Concern | Where it lives |
|---|---|
| Retry/backoff (external calls) | `retry.py` ā€” decorator on Anthropic, REST GitHub, Slack webhook, Datadog |
| Structured JSON logging with `incident_id`/`trace_id` correlation | `logging_config.py` ā€” ContextVar-based, formatter attaches automatically |
| OpenTelemetry (optional) | `telemetry.py` ā€” auto-instruments FastAPI + httpx if `OTEL_EXPORTER_OTLP_ENDPOINT` is set, no-op otherwise |
| HMAC webhook auth | `security.py` ā€” Datadog (base64), PagerDuty (multi-key `v1=`), generic (hex) |
| Rate limiting | `rate_limit.py` ā€” sliding window per `(ip, service)`, injectable clock |
| Async worker (fast 202) | `queue.py` ā€” `asyncio.Queue` + worker started/stopped via FastAPI lifespan |
| Alert deduplication | `dedup.py` ā€” bounded LRU with TTL, fingerprint per `(service, metric, severity, bucket)` |
| Runbook remediation | `executor.py` ā€” dry-run by default, `ShellExecutor` allow-list opt-in |
| Historical post-mortem RAG for triage | `history.py` ā€” service + keyword + recency scoring, injected as few-shot into the triage prompt |
| Streaming Slack brief updates | `agents/brief.py::compose_streaming_brief` + `_stream_triage` ā€” initial placeholder, then `chat.update` as each agent finishes (bot-token mode only) |
| Post-remediation verification loop | `verification.py` ā€” polls error rate after auto-execution, posts recovered / improving / still-elevated to the incident thread |

## Notes for the operator

- **Restart safety**: SQLite is written after each step, but the in-memory queue is *not* durable ā€” a hard kill mid-triage loses queued alerts. Swap `queue.py` for arq/Celery/Dramatiq (same interface) for at-least-once durability.
- **Multi-instance**: rate limiter and dedup index are in-memory. Back them with Redis for horizontal scaling.
- **Cost control**: three Claude calls per incident (triage, runbook, impact) plus one at resolve (post-mortem). At Sonnet 4.6 pricing that's ~$0.05ā€“0.15 per incident depending on runbook library size.
- **Post-mortem generator failure mode**: if Claude returns malformed JSON, `agents/postmortem.py` falls back to a deterministic template instead of dropping the post-mortem entirely.
- **What it doesn't do (yet)**: no incident merging across services, no on-call rotation lookup, no Jira/Linear ticket creation, no PR-annotation for the suspect commit. Each of those is a self-contained integration behind the existing adapter pattern.
